## Acceptance Criteria

1. **AC-1** — `Dashboard.tsx` **não importa mais** dos arquivos `@/data/{matches,stadiums,teams}.ts` (que são fixtures locais).
2. **AC-2** — Stats cards (Vendas Totais, Ingressos Vendidos, Usuários, Taxa de Conversão) consomem `api.getAdminStats()`.
3. **AC-3** — Lista "Vendas Recentes" mostra as últimas 5 vendas reais via `api.getSales()` (ordenadas por `created_at desc`, mostra `user_name`, `home_team x away_team`, `total_price`, `quantity`).
4. **AC-4** — Lista "Jogos Mais Vendidos" mostra os top 4 jogos por quantidade total vendida, derivado client-side de `api.getSales()` (agrupa por match_id, soma quantities, ordena desc).
5. **AC-5** — Quick stats (Jogos Fase de Grupos, Estádios, Seleções Classificadas) consomem `api.getMatches()`, `api.getStadiums()`, `api.getTeams()`.
6. **AC-6** — Indicadores de "growth %" (salesGrowth, ticketsGrowth, etc.) que não têm como ser computados honestamente são **removidos** do dashboard (não inventar trends — Constitution Article IV).
7. **AC-7** — Loading state visível enquanto fetch está em andamento (skeleton ou spinner por card).
8. **AC-8** — Error state se algum fetch falhar (toast + valor "—" no card afetado).
9. **AC-9** — Smoke test: `/admin` aberto → todos os cards mostram números **diferentes** dos mocks atuais (que vinham com $125.750, 847 ingressos, etc.).

### 1. Refatorar Dashboard.tsx para usar react-query (15 min)

- [ ] Substituir imports estáticos por imports do `api` client e `useQuery`:

```tsx
import React from 'react';
import { useQuery } from '@tanstack/react-query';
import {
  Ticket, Users, DollarSign, TrendingUp, Calendar, MapPin, Trophy,
} from 'lucide-react';
import { Card, CardContent, CardHeader, CardTitle } from '@/components/ui/card';
import { Skeleton } from '@/components/ui/skeleton';
import api from '@/lib/api';

const Dashboard: React.FC = () => {
  const { data: statsData, isLoading: statsLoading } = useQuery({
    queryKey: ['admin', 'stats'],
    queryFn: () => api.getAdminStats(),
  });

  const { data: salesData, isLoading: salesLoading } = useQuery({
    queryKey: ['admin', 'sales'],
    queryFn: () => api.getSales(),
  });

  const { data: matchesData } = useQuery({
    queryKey: ['matches'],
    queryFn: () => api.getMatches(),
  });

  const { data: stadiumsData } = useQuery({
    queryKey: ['stadiums'],
    queryFn: () => api.getStadiums(),
  });

  const { data: teamsData } = useQuery({
    queryKey: ['teams'],
    queryFn: () => api.getTeams(),
  });

  const stats = statsData?.data?.stats;
  const allSales = salesData?.data?.sales || [];
  const matches = matchesData?.data?.matches || [];
  const stadiums = stadiumsData?.data?.stadiums || [];
  const teams = teamsData?.data?.teams || [];

  // Últimas 5 vendas (já vem ordenado desc do backend)
  const recentSales = allSales.slice(0, 5);

  // Top 4 jogos por quantidade vendida
  const topMatches = React.useMemo(() => {
    const aggregated = new Map<number, { match: string; sold: number }>();
    for (const s of allSales) {
      const key = s.match_id;
      const label = `${s.home_team_code || s.home_team} x ${s.away_team_code || s.away_team}`;
      const cur = aggregated.get(key) || { match: label, sold: 0 };
      cur.sold += s.quantity;
      aggregated.set(key, cur);
    }
    const totalTicketsSold = stats?.total_tickets_sold || 1;
    return Array.from(aggregated.values())
      .sort((a, b) => b.sold - a.sold)
      .slice(0, 4)
      .map(m => ({ ...m, percentage: Math.min(100, Math.round((m.sold / totalTicketsSold) * 100)) }));
  }, [allSales, stats]);

  const groupPhaseMatches = matches.filter((m: any) => m.stage === 'Fase de Grupos').length;
  const classifiedTeams = teams.filter((t: any) => t.group_name).length;
  // ... rest of JSX
};
```

## Dev Notes

**Por que remover os "+X% este mês"?**
- Os números atuais (+23.5%, +18.2%, etc.) são totalmente inventados
- Computar trend real exigiria endpoint novo no backend (ex.: `GET /api/admin/stats?period=30d`) que retorna comparação period-over-period
- Para o evento, é mais honesto remover do que mostrar número falso
- Se quiser implementar trend de verdade depois, abre uma Story 0.5

**Por que `useQuery` e não `useEffect`?**
- O projeto já usa `@tanstack/react-query` (vide `App.tsx:33`)
- `useQuery` dá cache, retry, loading/error state out-of-the-box
- Múltiplas instâncias do dashboard compartilham o mesmo cache (boom de performance)

**Cuidado com tipos:**
- O backend retorna `recordset` do mssql sem tipagem TypeScript
- O `api.ts` tipa como `any[]` na maioria dos endpoints (ver lib/api.ts)
- Por isso usei `(s: any)` nos maps — TypeScript vai reclamar se forçar tipo. Pode tipar depois (story de polish), mas para esta story manter `any` é pragmático

**Sobre o `topMatches` derivação:**
- A query `/api/admin/sales` retorna **todas** as vendas (sem paginação no momento)
- Para 18 vendas (bacpac), client-side é trivial
- Se em produção o volume crescer (~10k+ vendas), considerar criar endpoint dedicado `GET /api/admin/top-matches?limit=4` no backend

**Manutenção do `recentSales` ordenação:**
- O backend já ordena `ORDER BY p.created_at DESC` (vide `routes/admin.js:51`), então `slice(0, 5)` no client pega corretamente os 5 mais recentes
