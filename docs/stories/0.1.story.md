## Story

**Como** Raphael (owner do projeto),
**eu quero** colocar a aplicação FIFA 2026 rodando em **Azure Web Apps + Azure SQL Database** desde o início,
**para que** eu tenha um ambiente de iteração rápida (build + deploy em minutos) para fazer os ajustes de conteúdo necessários antes do evento TFTEC, e qualquer mudança seja validada ponta-a-ponta em produção real do PaaS.

## Acceptance Criteria

1. **AC-1** — Resource Group `fifa2026-rg` criado em `eastus` com: 1 App Service Plan B1, 2 Web Apps Windows (`fifa2026-web` Node 20, `fifa2026-back` Node 18), 1 Azure SQL Server + Database (Basic).
2. **AC-2** — Banco `FIFA2026Tickets` populado com dados do `FIFA2026Tickets.bacpac` (16 seleções, 9 estádios, 12 jogos, 84 categorias, 10 usuários, 18 compras).
3. **AC-3** — Backend `fifa2026-back` deployado com `fifa2026-api/` e App Settings preenchidas (`DB_SERVER`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`, `JWT_SECRET`, `JWT_EXPIRES_IN`, `FRONTEND_URL`).
4. **AC-4** — Frontend `fifa2026-web` deployado com `dist/` gerado a partir de `BACKEND_URL=https://fifa2026-back.azurewebsites.net` (placeholder substituído no `web.config`).
5. **AC-5** — `https://fifa2026-web.azurewebsites.net` carrega a SPA, login com `admin@fifa2026.com / admin123` funciona, listagem de jogos retorna 12 jogos.
6. **AC-6** — Backend hardened: Access Restriction com allowlist dos outbound IPs do front + `defaultAction=Deny` (chamadas diretas de fora do Azure resultam em 403).
7. **AC-7** — `az group show -n fifa2026-rg --query tags` (ou equivalente) mostra que tudo é o estado-alvo descrito no EPIC-000.

- [ ] Garantir login: `az login && az account show`
- [ ] Definir secrets locais (não commitar):
  ```bash
  export SQL_ADMIN_PASSWORD='SuaSenh@Forte!'

  ```
- [ ] Criar RG: `az group create -n fifa2026-rg -l eastus`
- [ ] **Opção A (declarativo):** rodar Bicep
  ```bash
  cd infra
  az deployment group create \
    -g fifa2026-rg \
    --template-file main.bicep \
    --parameters parameters/dev.bicepparam \
    --name fifa2026-initial
  ```
- [ ] **Opção B (imperativo, didático):** rodar `bash provision.sh` ou `pwsh provision.ps1` (faz o mesmo passo-a-passo + já aplica Access Restrictions no passo 6)

### 2. Importar bacpac no Azure SQL (8 min)

- [ ] Subir bacpac para storage temporário e importar (script no `infra/README.md` seção "Importar o `.bacpac`"):
  ```bash
  STORAGE=fifa2026sa$RANDOM
  az storage account create -g fifa2026-rg -n $STORAGE -l eastus --sku Standard_LRS
  KEY=$(az storage account keys list -g fifa2026-rg -n $STORAGE --query [0].value -o tsv)
  az storage container create --name bacpac --account-name $STORAGE --account-key $KEY
  az storage blob upload \
    --account-name $STORAGE --account-key $KEY \
    --container-name bacpac --file FIFA2026Tickets.bacpac \
    --name FIFA2026Tickets.bacpac

  az sql db import \
    -g fifa2026-rg -s fifa2026-sql -n FIFA2026Tickets \
    --storage-key-type StorageAccessKey --storage-key $KEY \
    --storage-uri "https://$STORAGE.blob.core.windows.net/bacpac/FIFA2026Tickets.bacpac" \
    --admin-user fifa2026admin --admin-password "$SQL_ADMIN_PASSWORD"

  az storage account delete -g fifa2026-rg -n $STORAGE --yes
  ```
- [ ] Validar:
  ```bash
  sqlcmd -S fifa2026-sql.database.windows.net -d FIFA2026Tickets \
    -U fifa2026admin -P "$SQL_ADMIN_PASSWORD" \
    -Q "SELECT COUNT(*) FROM teams; SELECT COUNT(*) FROM matches;"
  # 16 / 12
  ```

## Dev Notes

**Por que PaaS desde o começo (e não VMs)?**
- Build + deploy em PaaS: ~3 minutos (zip → push → IIS recicla)
- Build + deploy em VM: ~30 minutos (provisionar OS, instalar IIS+iisnode, copiar arquivos, configurar)
- Para fazer ajustes iterativos na app, PaaS é ordem de magnitude mais rápido
- VMs ficam para a story didática do evento (EPIC-001), quando a app já estiver pronta

**Custo do ambiente desta story:**
- App Service Plan B1 (1 unidade compartilhada para 2 Web Apps): ~$13/mês
- Azure SQL Basic (5 DTUs): ~$5/mês
- **Total: ~$18/mês**, prorated por hora. Apagar com `az group delete --name fifa2026-rg --yes` zera o custo.

**Reutilização de artefatos:**
- 100% dos arquivos de infra já estão prontos em `infra/`
- 100% dos workflows de CI/CD já estão prontos em `.github/workflows/`
- 100% do build do frontend já é parametrizado por `BACKEND_URL`
- O backend já escuta em `0.0.0.0` via `HOST` env (irrelevante em iisnode mas útil para portabilidade)

Esta story basicamente **executa** o que já foi construído. Não há código novo necessário.

## Validation / Smoke Test

```bash
FRONTEND_URL="https://fifa2026-web.azurewebsites.net"
BACKEND_URL="https://fifa2026-back.azurewebsites.net"

# 1. Frontend carrega
curl -fsS -I "$FRONTEND_URL" | grep "200 OK"

# 2. Proxy /api funciona (via web.config -> backend)
curl -fsS "$FRONTEND_URL/api/health" | grep '"ok"'

# 3. Backend conecta no banco
curl -fsS "$BACKEND_URL/api/health/db" | grep '"connected"'

# 4. Login + count de jogos
TOKEN=$(curl -fsS -X POST "$FRONTEND_URL/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@fifa2026.com","password":"admin123"}' \
  | python -c "import sys, json; print(json.load(sys.stdin)['token'])")

MATCH_COUNT=$(curl -fsS "$FRONTEND_URL/api/matches" \
  -H "Authorization: Bearer $TOKEN" \
  | python -c "import sys, json; print(len(json.load(sys.stdin)['matches']))")

[ "$MATCH_COUNT" = "12" ] && echo "OK" || echo "FAIL: esperado 12, obtido $MATCH_COUNT"

# 5. Backend hardened (chamada direta deve dar 403 por Access Restriction default Deny)
HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" "$BACKEND_URL/api/health")
[ "$HTTP_CODE" = "403" ] && echo "OK: backend privado" || echo "AVISO: backend respondeu $HTTP_CODE (esperado 403 se Access Restriction está aplicada)"
```
