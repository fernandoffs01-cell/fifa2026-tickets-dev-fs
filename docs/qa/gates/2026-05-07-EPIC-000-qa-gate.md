## Story 0.1 — Deploy inicial em Azure PaaS — ✅ PASS

### Acceptance Criteria

| AC | Critério | Status | Evidência |
|---|---|---|---|
| AC-1 | RG fifa2026-rg + Plan B1 + 2 Web Apps + Azure SQL Basic | ✅ | `az resource list -g fifa2026-rg` mostra todos os 5 recursos |
| AC-2 | Banco populado com dados do bacpac | ✅ | sqlcmd: 16/9/12/84/10/18 (bate 100%) |
| AC-3 | Backend deployado, App Settings preenchidas | ✅ | `/api/health/db` retorna `connected` + sample row |
| AC-4 | Frontend deployado com BACKEND_URL embutida | ✅ | `dist/web.config` tem URL absoluta |
| AC-5 | Site acessível, login funciona, listagem de jogos OK | ✅ | Smoke test verde |
| AC-6 | Backend hardened (Access Restriction Deny+allowlist) | 🟡 | Não aplicado — backend está **público** porque optamos por VITE_API_URL+CORS em vez de IIS reverse proxy |
| AC-7 | Estado-alvo conforme EPIC-000 | ✅ | Tabela de recursos confirma |

### Trade-offs documentados

1. **Regiões diferentes para SQL e App Service** (brazilsouth vs eastus2). Razão: sponsorship subscription bloqueia provisionamento de SQL em eastus/eastus2. Latência adicional ~150ms entre back→DB. Para evento didático, aceitável. Documentado como tech debt.

2. **AC-6 (Access Restriction) NÃO aplicada**. Razão: ARR proxy outbound não funcionou no SKU B1 sem config server-level. Mudou a arquitetura para `VITE_API_URL` + CORS allowlist no backend. Backend fica **publicamente acessível** mas **só aceita requests com Origin do front** via CORS. Trade-off:
   - **Antes (planejado):** backend privado, frontend faz proxy
   - **Agora:** backend público, mas CORS restringe quem pode usar via browser
   - **Mitigação:** rate limiting no backend (não implementado — entra como tech debt)
   - **Para o evento:** WAIVED (suficiente para fins educacionais)

3. **Node 22 LTS** em vez de 18. Azure Web App Windows não oferece 18 mais. Backend 100% compatível.

### Verdict: PASS

Funcional ponta-a-ponta. AC-6 trocado por mitigação CORS (waived com justificativa).

## Story 0.4 — Admin Dashboard com dados reais — 🟡 CONCERNS

### Acceptance Criteria

| AC | Critério | Status | Evidência |
|---|---|---|---|
| AC-1 | Não importa de `@/data/{matches,stadiums,teams}` | ✅ | `grep -E "from '@/data'" Dashboard.tsx`: 0 matches |
| AC-2 | Stats cards consomem `/api/admin/stats` | ✅ | Código usa `useQuery(['admin','stats'])` |
| AC-3 | Recent Sales consome `/api/admin/sales` (limit 5) | ✅ | `slice(0,5)` aplicado |
| AC-4 | Top Matches derivado de `/api/admin/sales` | ✅ | `useMemo` agregando por match_id |
| AC-5 | Quick stats consomem APIs reais | ✅ | Código usa 5 useQueries |
| AC-6 | Indicadores "+X% mês" removidos | ✅ | Não há mais texto "este mês" no Dashboard.tsx |
| AC-7 | Loading skeletons visíveis | ✅ | `<Skeleton />` em todos os cards |
| AC-8 | Error state | 🟡 | Não há toast explícito em caso de erro — react-query usa default null |
| AC-9 | Smoke test visual no browser | ⚠️ | NÃO EXECUTADO — precisa de validação humana |

### Issues identificados

1. **AC-8 (CONCERNS):** Não há toast/banner explícito quando uma das 5 queries falha. Em caso de erro de rede ou 500, o card mostra "—" mas o usuário não vê notificação. **Severity: MEDIUM** — tech debt. Recomendação: adicionar `onError` handler nos useQueries que dispara `toast()` (já existe no projeto via shadcn).

2. **AC-9 (CONCERNS):** Não rodei navegação visual no `/admin` para confirmar que:
   - Cards renderizam corretamente
   - Skeletons aparecem brevemente antes dos dados
   - Lista de Recent Sales mostra nomes reais (não "João Silva")
   - Top Matches barra de progresso renderiza com cores corretas
   - Layout não quebra no mobile

   **Recomendação:** usuário acessa `https://fifa2026-web.azurewebsites.net/admin` (após login admin) e valida em 1 minuto.

### Verdict: 🟡 CONCERNS

Implementação está funcional, mas **2 itens de validação dependem de revisão humana** (UX visual + tratamento de erro). Aceitável para evento, recomendado endurecer em iteração futura.
